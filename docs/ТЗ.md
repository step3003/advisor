# Техническое задание — «Advisor» (личный финансовый планировщик)

**Версия:** 2.0 (пивот на клиент-серверную web-архитектуру)
**Дата:** 2026-07-20
**Автор:** бизнес-аналитик (согласовано с заказчиком)

> **Изменение относительно v1.0.** Первая версия проектировалась как офлайн-приложение
> на Go+Fyne с источником правды в папке файлов, синхронизируемой через iCloud.
> После уточнения сценария (устройства: **iPhone + Mac** для работы, **Android** — куда
> приходят банковские SMS; заказчик готов поднять **свой VPS**) архитектура изменена на
> **клиент-серверную web-архитектуру**. Причина: iCloud не покрывает Android, а
> автоматический захват банковских SMS возможен только на Android нативно. Общий сервер
> (VPS) стал единым хабом данных для всех устройств.
>
> **Что переиспользуется из v1.0:** слои `domain/*` и `application/*` (ядро на Go, покрыто
> тестами) переезжают на сервер почти без изменений; `infrastructure/index/sqlite` (БД),
> `infrastructure/nbrb` (курсы). **Что выведено из проекта:** `presentation/*` (Fyne UI) и
> `infrastructure/vault` (файлы под iCloud) — заменяются web-фронтом и серверной БД.

---

## 1. Цель и суть продукта

Кроссплатформенный личный финансовый инструмент, который позволяет:
1. Вносить **план** трат на календарный месяц с детализацией по категориям.
2. Учитывать **фактические** траты и доходы, минимизируя ручной ввод.
3. Сравнивать **план vs факт**, видеть остаток и отклонения.
4. Смотреть **отчёты и графики** за произвольный период (день … годы).

Доступен на iPhone, Mac и Android через **веб/PWA** (браузер или установка на домашний
экран). Банковские SMS на Android автоматически превращаются в операции.

## 2. Ключевые решения (согласованы с заказчиком)

| Параметр | Решение |
|---|---|
| Архитектура | **Клиент-сервер.** Go-бэкенд на VPS переиспользует ядро (`domain` + `application`) |
| Источник правды | **Серверная БД (SQLite)** на VPS — единый для всех устройств |
| Клиент | **Web/PWA (SPA):** React + TypeScript + Vite + **Mantine** (UI-компоненты) + **Recharts** (графики) |
| Бэкенд | **Go, HTTP JSON API**, чистая слоистая архитектура |
| Авторизация | Простая токен-авторизация (личное приложение, один пользователь/семья) |
| Захват SMS | Отдельное нативное **Android-приложение** (Kotlin), читает банковские SMS → POST на API |
| Синхронизация | Через сервер (единый источник правды); клиенты — тонкие, PWA кэширует |
| Язык интерфейса | **Только русский** |
| Валюты | **Мультивалютность**, базовая валюта — **BYN** |
| Источник курсов | **API Национального банка РБ** (`api.nbrb.by`), кэш на сервере |
| Категории | Двухуровневые (категория → подкатегория), предустановленные + пользовательские |
| Период планирования | Календарный месяц |
| Период статистики | Произвольный (от дня до нескольких лет) |
| Развёртывание | Свой VPS: systemd + TLS (Caddy), автоматические бэкапы БД |

## 3. Стратегия минимизации ручного ввода

Полностью автоматический захват трат в реальном времени на iPhone невозможен (Apple
запрещает чтение SMS; облачные банковские агрегаторы ломают приватность и слабо покрывают
РБ). Поэтому усилие расслаивается:

- **Слой 1 — предсказуемое на автопилоте:** повторяющиеся шаблоны (зарплата, аренда,
  подписки) подставляются без участия пользователя (FR-REC).
- **Слой 2 — пачкой:** импорт банковской выписки (CSV) + **авто-категоризация по правилам**
  («продавец → категория»), запоминаемым между импортами (FR-IMPORT).
- **Слой 3 — автозахват SMS:** Android-приложение ловит банковские SMS и создаёт операции
  автоматически (FR-SMS).
- **Слой 4 — быстрый ручной ввод:** для наличных/разового — «повторить прошлую» в один тап,
  умные значения по умолчанию; опционально скан QR-чека (в бэклоге).

## 4. Функциональные требования

### 4.1. Категории (FR-CAT)
- **FR-CAT-1.** Двухуровневая структура: категория → подкатегория (подкатегория опциональна).
- **FR-CAT-2.** Предустановленный набор при инициализации (Приложение A). Тип: `expense`/`income`.
- **FR-CAT-3.** Создание/переименование/архивирование своих категорий и подкатегорий.
- **FR-CAT-4.** Жёсткое удаление запрещено при наличии операций/планов — только архивирование.
- **FR-CAT-5.** Поля категории: название, тип, иконка (опц.), цвет (для графиков, опц.).

### 4.2. Планирование (FR-PLAN)
- **FR-PLAN-1.** Планы на календарный месяц (`год-месяц`).
- **FR-PLAN-2.** Плановая позиция: категория/подкатегория + сумма + валюта + комментарий.
- **FR-PLAN-3.** Уникальность позиции по (категория, подкатегория, валюта) в месяце.
- **FR-PLAN-4.** Копирование плана прошлого месяца в текущий.
- **FR-PLAN-5.** Плановый доход (категории типа `income`) — аналогично.

### 4.3. Фактические операции (FR-TX)
- **FR-TX-1.** Транзакция: дата, тип, категория/подкатегория, сумма, валюта, комментарий, источник (ручная/SMS/импорт/повтор).
- **FR-TX-2.** Создание/редактирование/удаление.
- **FR-TX-3.** Сумма в исходной валюте; агрегаты — пересчёт в BYN по курсу НБ РБ на дату операции.
- **FR-TX-4.** Быстрый ввод: последние категории, значения по умолчанию (сегодня, валюта по умолчанию), «повторить прошлую».

### 4.4. План vs Факт и баланс (FR-BAL)
- **FR-BAL-1.** По каждой категории за месяц: план, факт, отклонение, % исполнения.
- **FR-BAL-2.** Итоги: плановый/фактический доход и расход, остаток = доход_факт − расход_факт.
- **FR-BAL-3.** «Осталось до конца месяца» (по категориям и в целом), индикация перерасхода.
- **FR-BAL-4.** Агрегаты в BYN + разбивка по исходным валютам, если операции в разных валютах.

### 4.5. Повторяющиеся траты (FR-REC)
- **FR-REC-1.** Шаблон: категория, сумма, валюта, ежемесячная периодичность, день месяца, дата начала, дата окончания (опц.), тип, флаг «автосоздавать факт».
- **FR-REC-2.** При наступлении нового месяца шаблоны идемпотентно подставляются в план.
- **FR-REC-3.** Опционально создают и фактическую операцию (по флагу).
- **FR-REC-4.** Управление: создать/изменить/приостановить/удалить.

### 4.6. Отчёты и графики (FR-REP)
- **FR-REP-1.** Произвольный период: пресеты (день/неделя/месяц/квартал/год) + «с…по…».
- **FR-REP-2.** Разбивка расходов по категориям (диаграмма).
- **FR-REP-3.** Динамика по месяцам: доход vs расход, план vs факт.
- **FR-REP-4.** Топ категорий по расходу.
- **FR-REP-5.** Суммы отчётов — в BYN (пересчёт по курсу на дату операции).

### 4.7. Мультивалютность и курсы (FR-CUR)
- **FR-CUR-1.** Справочник валют (мин. BYN, USD, EUR, RUB; расширяемо из перечня НБ РБ).
- **FR-CUR-2.** Базовая валюта — BYN (в MVP фиксирована).
- **FR-CUR-3.** Курсы с сервера: `GET https://api.nbrb.by/exrates/rates?ondate=YYYY-MM-DD&periodicity=0`, кэш в БД по дате. Учитывать `Cur_Scale`/`Cur_OfficialRate`.
- **FR-CUR-4.** При отсутствии курса на дату — последний доступный ≤ даты, помечать приблизительным.
- **FR-CUR-5.** Периодическое фоновое обновление курсов на сервере (cron/тикер).

### 4.8. Авторизация (FR-AUTH)
- **FR-AUTH-1.** Доступ к API только по токену (Bearer). Личное приложение: один аккаунт (или несколько для семьи в бэклоге).
- **FR-AUTH-2.** Токен для Android-форвардера — отдельный (device token), можно отозвать.
- **FR-AUTH-3.** Весь трафик по HTTPS (TLS на VPS).

### 4.9. Автозахват SMS (FR-SMS)
- **FR-SMS-1.** Нативное Android-приложение: `BroadcastReceiver` на `SMS_RECEIVED`, разрешение на чтение SMS.
- **FR-SMS-2.** Парсинг SMS по настраиваемым **шаблонам банков** (отправитель + regex суммы/валюты/операции). Форматы SMS банков РБ уточняются по реальным сообщениям.
- **FR-SMS-3.** Распознанная операция отправляется на API (`POST /transactions` с источником `sms`); при недоступности сети — локальная очередь и повтор.
- **FR-SMS-4.** Нераспознанные SMS не теряются: сохраняются как «черновики» на сервере для ручного разбора.
- **FR-SMS-5.** Распространение — сайдлоад APK (Google Play ограничивает доступ к SMS).

### 4.10. Импорт выписки и авто-категоризация (FR-IMPORT)
- **FR-IMPORT-1.** Импорт банковской выписки (CSV; форматы уточняются по реальным выпискам).
- **FR-IMPORT-2.** Правила авто-категоризации: сопоставление продавца/описания → категория, запоминаются и применяются к будущим импортам/SMS.
- **FR-IMPORT-3.** Дедупликация: повторный импорт не создаёт дублей (по дате+сумме+описанию/хешу).
- **FR-IMPORT-4.** Предпросмотр перед применением: пользователь подтверждает/правит категории.

### 4.11. Экспорт / бэкап (FR-IO)
- **FR-IO-1.** Экспорт снапшота всех данных в JSON (версионированный формат).
- **FR-IO-2.** Импорт снапшота: режимы «заменить всё» / «объединить» (по UUID).
- **FR-IO-3.** Экспорт операций за период в CSV.
- **FR-IO-4.** Автоматический серверный бэкап БД (по расписанию, с ротацией).

### 4.12. Настройки (FR-SET)
- **FR-SET-1.** Валюта по умолчанию для ввода.
- **FR-SET-2.** Управление категориями.
- **FR-SET-3.** Обновить курсы валют вручную.
- **FR-SET-4.** Экспорт/импорт данных.
- **FR-SET-5.** Управление шаблонами банковских SMS и правилами авто-категоризации.
- **FR-SET-6.** Управление токенами устройств (Android-форвардер).

## 5. Нефункциональные требования

- **NFR-1. Чистая архитектура (бэкенд).** Слои изолированы: Transport(HTTP) → Application → Domain; Infrastructure реализует порты. Домен не знает о HTTP/SQLite/сети.
- **NFR-2. Точность денег.** Никаких `float`. Минорные единицы `int64` (копейки) / десятичные строки.
- **NFR-3. Тестируемость.** Домен и application покрыты unit-тестами (≥70%); API — интеграционными тестами (httptest).
- **NFR-4. Приватность.** Данные только на **своём** VPS пользователя; наружу уходят лишь запросы курсов к api.nbrb.by (без персональных данных). Нет сторонних аналитик/трекеров.
- **NFR-5. PWA/UX.** Устанавливается на домашний экран (iOS/Android/Mac), адаптивная вёрстка (телефон/десктоп), офлайн-кэш прочитанного, современный аккуратный вид (Mantine).
- **NFR-6. Надёжность данных.** UUID у записей; версионируемые миграции БД; серверные бэкапы; идемпотентность автозахвата (SMS/импорт не плодят дубли).
- **NFR-7. Безопасность.** HTTPS обязателен; секреты/токены — вне репозитория (env); rate-limiting на публичных эндпоинтах.
- **NFR-8. Локализация.** Строки UI вынесены (ru), архитектура готова к добавлению языков.
- **NFR-9. Стоимость поддержки.** Единый Go-бэкенд + один SPA-фронт; минимум внешних зависимостей и сервисов.

### 6.1. Бэкенд (Go) — структура
```
server/                          # или адаптация текущего дерева
├── cmd/advisord/                # main: DI, конфиг (env), запуск HTTP-сервера
├── internal/
│   ├── domain/                  # ← ПЕРЕНОС из v1.0 без изменений
│   ├── application/             # ← ПЕРЕНОС из v1.0 (usecase-сервисы + ports)
│   ├── infrastructure/
│   │   ├── index/sqlite/        # ← ПЕРЕНОС: серверная БД (источник правды)
│   │   ├── nbrb/                # ← ПЕРЕНОС: клиент курсов
│   │   ├── auth/                # токены устройств/пользователя
│   │   └── importer/            # парсеры CSV-выписок + правила авто-категоризации

```
**Правила зависимостей:** `transport/http` → `application` → `domain`; `infrastructure` реализует порты. Домен без HTTP/SQL/сети.

```
web/
├── src/
│   ├── api/           # клиент к JSON API (fetch/axios), типы DTO
```

## 7. Модель данных (серверный SQLite)

Как в v1.0, но БД — единый источник правды на сервере (не индекс). `id` — UUID; деньги —
`amount_minor INTEGER` + `currency TEXT`. Основные таблицы: `categories`, `currencies`,
`exchange_rates`, `plan_items`, `transactions` (+ поле `source`: manual|sms|import|recurring),
`recurring_templates`, `settings`, `schema_migrations`. Добавляются:

```sql
-- Токены доступа (пользователь и устройства-форвардеры)
auth_tokens(id TEXT PK, name TEXT, token_hash TEXT, kind TEXT, created_at TEXT, revoked_at TEXT NULL)

-- Правила авто-категоризации (FR-IMPORT-2)
category_rules(id TEXT PK, match_type TEXT, pattern TEXT, category_id TEXT, priority INTEGER, created_at TEXT)

-- Шаблоны разбора банковских SMS (FR-SMS-2)
sms_templates(id TEXT PK, sender TEXT, regex TEXT, amount_group INTEGER, currency_group INTEGER,
              type TEXT, enabled INTEGER, created_at TEXT)

-- Черновики нераспознанных SMS/строк выписки (FR-SMS-4)
inbox_drafts(id TEXT PK, source TEXT, raw TEXT, received_at TEXT, resolved INTEGER)
```

Записи несут `updated_at`; для дедупликации импорта/SMS — стабильный хеш операции.

## 8. HTTP API (эскиз)

Все ответы/запросы — JSON; авторизация `Authorization: Bearer <token>`; ошибки — единый формат.

```
GET    /api/categories                 POST /api/categories        PATCH /api/categories/{id}
GET    /api/plans?ym=YYYY-MM           PUT  /api/plans             POST  /api/plans/copy-previous
GET    /api/transactions?ym=YYYY-MM    POST /api/transactions      PATCH /api/transactions/{id}   DELETE …
GET    /api/reports/plan-vs-fact?ym=…  GET  /api/reports/period?from=…&to=…
GET    /api/recurring                  POST /api/recurring         PATCH /api/recurring/{id}
POST   /api/currency/refresh           GET  /api/currency/rates
POST   /api/ingest/sms                 (для Android-форвардера; source=sms)
POST   /api/import/statement           (CSV; предпросмотр → подтверждение)
GET    /api/export/json                GET  /api/export/csv?from=…&to=…
POST   /api/auth/tokens                DELETE /api/auth/tokens/{id}
GET    /api/settings                   PATCH /api/settings
```

## 9. Развёртывание (VPS)

- Один статически слинкованный Go-бинарь (`advisord`) + встроенная раздача статики фронта (или отдельно Caddy).
- **Caddy** как reverse-proxy с автоматическим TLS (Let's Encrypt) на домене.
- **systemd**-юнит для автозапуска/рестарта.
- Конфиг через env (порт, путь к БД, секрет токенов, домен).
- **Бэкап**: периодический дамп SQLite (`VACUUM INTO`) с ротацией; выгрузка в отдельное хранилище опц.
- CI (опц.): сборка бинаря + фронта, деплой по SSH.

## 10. Границы MVP

**В MVP:** разделы 4.1–4.9, 4.11, 4.12; развёртывание (раздел 9); авто-категоризация базовая (4.10 — правила + SMS-маршрут). 
**Не в MVP (бэклог):** импорт PDF-выписок; скан QR-чеков и детализация позиций; много-пользовательский доступ (семья) с ролями; бюджеты-цели/накопления; пуш-уведомления; мультиязычный UI.

## 11. Критерии приёмки (ключевые)

1. Бэкенд собирается в один Go-бинарь; поднимается на VPS за Caddy по HTTPS; API отвечает только с валидным токеном.
2. Ядро (`domain`/`application`) из v1.0 переиспользовано; в домене нет импортов HTTP/SQL/сети; деньги нигде не во `float`; тесты ядра проходят.
3. Через web/PWA на iPhone, Mac и Android можно: завести категорию, задать план, внести факт, увидеть план/факт, остаток и «осталось до конца месяца».
4. Операции в разных валютах пересчитываются в BYN по курсу НБ РБ на дату; курсы кэшируются на сервере, обновляются вручную и по расписанию.
5. Повторяющийся шаблон один раз подставляется в план нового месяца (без дублей).
6. Отчёт за произвольный период отображает разбивку по категориям и динамику по месяцам (Recharts).
7. Android-форвардер: тестовое банковское SMS распознаётся по шаблону и создаёт операцию через `POST /api/ingest/sms`; нераспознанное попадает в черновики; повторная доставка не создаёт дубль.
8. Импорт CSV-выписки применяет правила авто-категоризации, показывает предпросмотр, не плодит дубли при повторном импорте.
9. PWA устанавливается на домашний экран и открывает ранее загруженные данные из кэша при кратком отсутствии сети.
10. Экспорт → импорт снапшота восстанавливает данные без потерь (round-trip); работает серверный бэкап БД.
