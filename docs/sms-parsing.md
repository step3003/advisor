# Разбор SMS — архитектура и правила (единый источник истины)

> Держать эту логику целиком. Любое изменение разбора SMS — сверяться сюда и обновлять этот файл.

## 1. Конвейер приёма SMS (`sms.Service.Ingest`)

```
SMS (sender, text, receivedAt)
        │
   parseWith  ── ДВА ПРОХОДА по шаблонам:
        │        1) сначала ВСЕ шаблоны-мусор (action=discard)  → фильтр приоритетнее
        │        2) затем шаблоны-операции (action=operation)
        ▼
 ┌─ совпал discard ──────────────► в архив «Отфильтрованные» (draft.status=filtered),
 │                                  контрагент НЕ создаётся, операции нет.
 │
 ├─ совпал operation ─────────────► Observe(признак) в справочник, затем категория:
 │        категория = признак.categoryId (точная привязка)
 │                  → иначе дефолт шаблона (DefaultCategoryID)
 │        есть категория → операция (ledger.AddFromSMS, merchant_key=признак)
 │        нет категории  → во «Входящие» (draft.status=pending), признак уже в справочнике
 │
 └─ ничего не совпало ────────────► во «Входящие» (нераспознанное)
```

**Ключевое правило: `discard` всегда бьёт `operation`.** Поэтому даже если широкий пресет
ловит сообщение как операцию, фильтр-мусор с более специфичным признаком отменит это.

## 2. Сущности

- **Template (тип сообщения)** `internal/application/sms/sms.go`
  - `Action`: `operation` | `discard`.
  - `operation`: `Pattern` + группы `AmountGroup/CurrencyGroup/MerchantGroup`, `CaptureKind` (merchant|account), `Type` (expense|income), `DefaultCategoryID`.
  - `discard`: только `Pattern` (литерал-признак мусора); полей не извлекает.
  - Шаблоны ГЛОБАЛЬНЫЕ (админ-пресеты для всех). Создаются «по образцу» или вручную.
- **MerchantEntry (признак)** — то, что ловит `MerchantGroup`:
  - `Kind`: `merchant` (контрагент — имя) | `account` (счёт — номер, ЕРИП/перевод).
  - `CategoryID` — точная привязка категории (per-owner).
  - `Label` — человеческое название (для счетов показывается в операции вместо номера).
  - Статистика (встреч/оборот) считается ВЖИВУЮ из операций по `transactions.merchant_key`.
- **Draft (черновик)** — входящая SMS: `status` = `pending` («Входящие») | `filtered` (архив).

## 3. Механизмы (как чем пользоваться)

| Задача | Механизм | API |
|---|---|---|
| Научить новый банк | «По образцу»: тыкаешь поля/тип | `POST /api/sms/templates/from-sample` |
| Выкинуть мусор (баланс/коды) | тип `discard` по слову-признаку | from-sample `action=discard` |
| Признак → категория | привязка к контрагенту/счёту | `POST /api/sms/merchants/assign` |
| **Не учитывать признак** | заводит discard на имя + убирает из справочника | `POST /api/sms/merchants/ignore` |
| Убрать ложный признак | удаление из справочника | `POST /api/sms/merchants/delete` |
| Вернуть из архива | статус → pending | `POST /api/inbox/{id}/restore` |

**«Не учитывать» (`IgnoreMerchant`)** = создать `discard`-шаблон с паттерном
`QuoteMeta(имя)` + удалить признак из справочника. Так как discard приоритетен,
все будущие сообщения с этим именем уходят в архив.

## 4. Рабочая схема (текущие правила разбора)

| Что | Пример текста | Как обрабатывать |
|---|---|---|
| Покупка картой | `Oplata 34.80 BYN. BLR MAGAZIN EVROOPT. Balance…` | operation, расход, признак=контрагент (`MAGAZIN EVROOPT`) → его категория |
| ЕРИП-платёж | `schet platezha 2162…*140. Summa 40.46 BYN…` | operation, расход, признак=счёт → его категория, ярлык-название |
| **Перевод между своими картами (исходящий)** | `Perevod 1.00 BYN. BLR P2P SDBO NO FEE. Balance…` | **discard** по `P2P SDBO NO FEE` → архив |
| **Зачисление ОТ СЕБЯ** | `Zachislenie perevoda 12.00 BYN. BLR ALIAKSEI KRAUTSOU…` | **discard** по своему имени (`ALIAKSEI KRAUTSOU`) → архив |
| **Зачисление ОТ ЧЕЛОВЕКА** | `Zachislenie perevoda 0.50 BYN. BLR ULYANA NARONSKAYA…` | operation, **доход**, признак=контрагент (имя) → доход |

Логика «своё vs чужое» для зачислений:
- шаблон «Зачисление перевода» = доход по умолчанию (любой отправитель);
- discard на СВОЁ имя перебивает его → зачисления от себя в архив;
- любая другая фамилия (кроме моей) → остаётся доходом.
То есть «любой, кроме меня → прибыль» реализуется как: `default income` + `discard(моё имя)`.

## 5. Подводные камни

- Пресет Приорбанка `(?:Oplata|Perevod|Snyatie)` ловит и **Perevod** — поэтому P2P без
  фильтра становится операцией с ложным контрагентом. Лечится discard-фильтром (он приоритетен).
- Токены из SMS приходят с прилипшей пунктуацией (`BYN.`, `NARONSKAYA.`). Синтез «по образцу»
  обрезает пунктуацию по краям выделенного (`trimFieldPunct`).
- Признак с пробелами (имя) НЕ ломает синтез: паттерн выбирается по содержимому
  (номер → `[0-9*]+`, иначе → `(.+?)` с якорем), а не по типу.
- Признак, помеченный «Не учитывать», НЕ должен пересоздаваться: discard-ветка не делает Observe.

## 6. Ключевые файлы

- `internal/application/sms/sms.go` — Service, Ingest, parseWith (discard-first), Assign/Ignore/Delete Merchant.
- `internal/application/sms/synth.go` — сборка шаблона «по образцу» (SynthesizeTemplate).
- `internal/infrastructure/index/sqlite/sms.go`, `merchants.go` — репозитории.
- `internal/transport/http/handlers_sms.go`, `server.go` — эндпоинты.
- `web/src/pages/SmsPage.tsx` — UI (Входящие, Отфильтрованные, Контрагенты/счета, Типы, TeachModal, ResolveModal).

## 7. Экраны UI (SMS-парсер)

1. **Входящие** — новое/нераспознанное. Кнопка «Разобрать»:
   нет суммы → «по образцу» (создать тип); есть сумма → категоризация + «Не учитывать (мусор)».
2. **Отфильтрованные** (свёрнуто) — архив мусора, помечен типом, «Вернуть».
3. **Контрагенты и счета** — категории/названия; иконки 🚫 «не учитывать» и 🗑 «удалить».
4. **Типы сообщений** (свёрнуто) — все шаблоны (operation/discard), правка/удаление.
